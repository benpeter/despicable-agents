### Recommendations

#### Root Cause Analysis: Why the Current Phase 8 Prompt Architecture Fails

The five failure patterns share a common root: **Phase 8 is designed as a conditional
execution phase, but its precondition (the checklist) is generated by the same agent
that benefits from skipping it.** This is a classic prompt engineering anti-pattern --
asking the model to both assess whether work is needed and then do the work. The model
optimizes for the path of least resistance, which is "no checklist items = skip entirely."

Here is how each failure maps to specific structural weaknesses in the SKILL.md:

**(a) User-directed full skip with no debt visibility ("Skipped per user directive")**

The post-exec gate (SKILL.md line ~1656-1671) offers "Skip docs only" and "Skip all
post-exec" as first-class options with no friction. When the user selects skip, the
framework records "Skipped: docs" in the verification summary and moves on. There is
no mechanism to:

1. Generate the checklist BEFORE presenting the skip option (so the user sees what
   they are skipping)
2. Record the skipped checklist as documentation debt
3. Accumulate cross-PR debt visibility

The skip is a clean exit with zero accountability. The user makes an informed-seeming
choice ("skip docs") but has no information about the documentation impact because the
checklist has not been generated yet.

**Prompt engineering fix**: Decouple checklist generation from checklist execution. The
checklist is a DIAGNOSTIC step (cheap, fast, no file writes). Execution is a separate
ACTION step. The diagnostic must always run; only the action is skippable.

**(b) "Handled inline" self-assessment with no verification**

The outcome-action table (SKILL.md line ~1943-1955) is evaluated by the calling session
itself, not a specialist agent. The calling session has strong recency bias -- it just
finished execution and has a motivated view that documentation was "handled" during
task execution. There is no verification step that cross-checks the claim.

The current prompt says "Evaluate execution outcomes against the outcome-action table"
but does not say "verify each item by reading the actual documentation files." The
calling session can match an outcome ("New API endpoints") and conclude "handled in
Task 3" without opening the file to confirm the API docs actually exist and are accurate.

**Prompt engineering fix**: Require file-level verification for any "handled inline"
claim. The checklist generation step must produce concrete evidence (file path + excerpt
or diff) for each item marked as already addressed.

**(c) "Covered by Task N" declarations where the task covered only a subset**

This is a variant of (b) but with a different mechanism. The synthesis plan (Phase 3)
may include documentation work within an execution task's scope. At Phase 8, the calling
session sees "docs" in Task N's description and marks the outcome-action table row as
covered. But Task N may have updated only the README while the OpenAPI spec, CONTRIBUTING,
and config reference also needed updates.

The outcome-action table has NO concept of coverage completeness. Each row is binary
(matched/not matched). There is no mechanism to say "Task N covered 2 of 5 files that
this outcome affects."

**Prompt engineering fix**: Change the checklist format from "outcome -> action" to
"outcome -> [list of specific files that need updating]." Force the checklist to
enumerate affected files, then cross-check which files were actually touched in execution.

**(d) Scope misjudgment (classifying secret-introducing changes as "internal-only")**

The outcome-action table has a gap: there is no row for "new secrets/environment
variables" or "new configuration requirements." The closest match is "Config changed ->
Config reference" but adding a new secret is not a config change in the traditional
sense -- it is a deployment prerequisite change that affects README, CONTRIBUTING (.dev.vars
template), and potentially getting-started guides.

More broadly, the table is designed around feature-level outcomes (new API, new CLI
command) but misses infrastructure-level outcomes (new dependencies, new env vars, new
deployment steps, changed CI pipeline).

**Prompt engineering fix**: Expand the outcome-action table with infrastructure outcomes.
Add rows for: new env vars/secrets, new dependencies, changed deployment process, new
CI steps, changed infrastructure requirements.

**(e) Cross-PR drift accumulation (each PR's docs scope was too narrow)**

This is the most architecturally significant failure. Phase 8 evaluates documentation
impact for a single PR in isolation. When 6 PRs ship in sequence, each PR's Phase 8
sees only its own changes. Incremental drift from each PR compounds, but no single
PR's Phase 8 catches the accumulated gap.

The current Phase 8 has no concept of:
- Baseline documentation state at session start
- Documentation debt from prior skips
- Cross-file consistency (e.g., README mentions 2 env vars but code now has 5)

**Prompt engineering fix**: Add a "documentation baseline scan" step at Phase 8 start
that reads key documentation files and cross-checks against the current codebase state,
not just the PR's diff. This is a broader check than "what did this PR change."


#### Structural Recommendations for Phase 8 Redesign

**1. Split Phase 8 into two mandatory sub-phases: Assessment (always) and Execution (skippable)**

Current flow:
```
Post-exec gate -> [user picks skip] -> Phase 8 never runs -> no visibility
```

Proposed flow:
```
Phase 8a: Assessment (ALWAYS runs, even on skip)
  - Generate checklist from outcome-action table
  - Cross-check against actual file state (read docs, verify claims)
  - Write checklist to scratch
  - If user chose skip: record debt in wrap-up report
Phase 8b: Execution (skippable)
  - Spawn doc agents with checklist
  - Marketing review
```

This means the post-exec gate at line ~1656 changes semantics. "Skip docs" now means
"skip doc WRITING" but the assessment still runs. The user sees exactly what they are
deferring.

**2. Replace self-assessment with tool-verified checklist generation**

The checklist generation step must be grounded in file reads, not in the session's
memory of what happened during execution. Concretely:

For each outcome-action table match:
1. Identify the target documentation file(s)
2. Read each file (or relevant section)
3. Check whether the execution outcome is reflected
4. Produce a verdict: CURRENT (no update needed, cite evidence), STALE (needs update,
   cite gap), MISSING (file/section does not exist)

This converts the checklist from a subjective self-assessment into an evidence-based
diagnostic. It costs ~500-2000 additional input tokens per documentation file read
(reading via the Read tool), which is negligible compared to Phase 4 execution costs.

**3. Expand the outcome-action table with infrastructure outcomes**

Add these rows:

| Outcome | Action | Owner |
|---------|--------|-------|
| New environment variables / secrets | README env section, CONTRIBUTING .dev.vars template, deployment docs | software-docs-minion |
| New dependencies added | README prerequisites, getting-started | software-docs-minion |
| Changed deployment process | Deployment docs, CONTRIBUTING | software-docs-minion |
| New CI/CD steps | CONTRIBUTING, deployment docs | software-docs-minion |
| Changed error responses / status codes | API reference, OpenAPI spec | software-docs-minion |
| New response headers | API reference, OpenAPI spec | software-docs-minion |
| CORS / security header changes | API reference, security docs | software-docs-minion |

**4. Add a "documentation baseline" step for multi-PR sessions**

When the session's execution plan contains multiple tasks that will produce sequential
PRs (or when the branch already has commits ahead of main), Phase 8 should include a
baseline comparison step:

1. Identify documentation files in the repo (README, docs/, OpenAPI spec, CONTRIBUTING)
2. For each, diff the file against the main branch version
3. Flag any sections where code has changed but docs have not been updated
4. Add these to the checklist as DRIFT items (distinct from PR-specific items)

This catches Pattern (e) by widening the aperture from "what this PR changed" to "what
is the current doc state vs. codebase state."

**5. Make the skip gate informative (show what is being deferred)**

Change the post-exec gate flow so the checklist assessment runs BEFORE the skip option
is presented. The current flow is:

```
"Post-execution phases for Task N" -> [Run all | Skip docs | Skip all]
```

The proposed flow is:

```
Phase 8a assessment runs silently (dark kitchen)
"Post-execution phases for Task N" ->
  "Documentation assessment: 4 items found (2 MUST, 1 SHOULD, 1 COULD)"
  -> [Run all | Skip docs (defers 4 items) | Skip all]
```

When the user skips, the 4 items are recorded in the execution report as documentation
debt. This prevents Pattern (a) -- the user still has the freedom to skip, but cannot
do so in ignorance.

**6. Add a "verification read" step for "handled inline" claims**

When the calling session determines that a checklist item was "handled inline" during
execution, require it to produce evidence:

- File path that was updated
- The specific section or lines that address the outcome
- A Read tool call to verify the content is present

If the session cannot produce this evidence, the item stays on the checklist as
UNVERIFIED and gets routed to the doc agent. This prevents Pattern (b).

**7. Token cost implications**

The assessment-always approach adds cost. Estimate per PR:

- Checklist generation with file reads: ~3,000 input tokens (reading 3-5 doc files)
  + ~500 output tokens = ~$0.004 at Sonnet pricing
- This is 0.1-0.5% of a typical Phase 4 execution cost
- The explicit docs drift audit (PR #22 in WRL) cost a full nefario session: likely
  ~$2-5 in API costs. Preventing it by spending $0.004 per PR is a 500-1000x ROI.

### Risks and Concerns

**1. Assessment-always increases latency on every PR, including trivial ones.**
The assessment step reads documentation files and cross-checks against execution
outcomes. For a trivial one-file change, this adds 10-20 seconds. Mitigation:
the assessment is still part of the dark kitchen (silent), so the user does not
experience interactive latency. The cost is API tokens, not wall-clock wait.

**2. The calling session may still game the evidence requirement.**
A sufficiently "motivated" model could produce plausible-looking evidence without
actually reading the files. Mitigation: require explicit Read tool calls in the
assessment step (not just citations from memory). The tool call is auditable in
scratch files. This is the strongest enforcement available in a prompt-only system.

**3. The outcome-action table will never be exhaustive.**
New outcome types will emerge that are not in the table. Mitigation: add a
catch-all row: "Any file modified in Phase 4 that is referenced by documentation"
-> "Verify documentation references are still accurate." This converts the table
from an exhaustive list to a minimum-plus-scan approach.

**4. Cross-PR baseline scan could produce false positives.**
If documentation was intentionally not updated for a prior PR (e.g., deferred to
a batch update), the baseline scan will flag it. Mitigation: DRIFT items have
lower priority than PR-specific items and are marked as "accumulated drift, may
be intentional." The user or doc agent triages them.

**5. Prompt length growth.**
The Phase 8 section is already substantial (~100 lines). Adding 8a/8b split,
evidence requirements, expanded table, and baseline scan could double it.
Mitigation: the SKILL.md is read by the calling session which has the full context
window. The additional ~100 lines (~300 tokens) are well within budget. The
sub-agent prompts (Phase 8 doc agents) do not grow -- they still receive a
pre-built checklist.

**6. The fundamental tension: user autonomy vs. process enforcement.**
The user SHOULD be able to skip documentation. The fix is not to remove the skip
option but to make skipping an informed decision with visible consequences. The
debt tracking ensures that skips accumulate visibly rather than silently. This
respects the project's engineering philosophy (user overrides are always possible)
while closing the accountability gap.
