## Domain Plan Contribution: software-docs-minion

### Root Cause Analysis

The Phase 8 outcome-action table has two structural weaknesses that caused the WRL drift:

**1. The taxonomy is additive-only; it detects new things but not changed things.**

The table rows -- "New API endpoints", "Config changed", "Architecture changed" -- all describe additions or high-level changes. They miss the most common drift pattern: existing documentation contradicting modified behavior. In the WRL case:

- Key rotation behavior changed (PR #54), but the README still contained the old warning about rotation breaking verification. No outcome row catches "behavior of existing feature changed."
- POST error responses gained CORS headers (PR #57), but existing OpenAPI error response definitions lacked them. No outcome row catches "existing response shape changed."
- The health endpoint was added as part of infrastructure work, classified by the orchestrator as "internal" -- but it exposed a public HTTP endpoint with legal-required content. No outcome row catches "new publicly accessible endpoint added as side effect of infrastructure task."

**2. The table operates on Phase 4 execution outcomes, not on file-level diff analysis.**

The checklist is generated by evaluating "execution outcomes" -- a semantic summary of what happened. This relies on the orchestrator accurately classifying what happened. When the orchestrator misclassifies (Pattern 4: "internal logging change" for a PR that introduced IP_HASH_SEED), the wrong rows match. A diff-based approach -- scanning what files actually changed and what symbols were introduced -- would catch drift that semantic classification misses.

**3. Phase 4 documentation task completeness is never verified.**

When a Phase 4 execution task includes documentation (Pattern 3: "Documentation covered by Task 3"), Phase 8 treats the existence of a docs task as proof of completeness. There is no verification step that compares what the docs task updated against what actually needed updating. Task 3 in PR #51 updated cleanup documentation but missed the Key Archive endpoint, 3 OpenAPI discrepancies, and README secret references.

### Recommendations

#### R1: Expand the outcome-action table with six missing outcome categories

The current 10-row table needs to grow to cover the drift patterns actually observed. Proposed additions:

| Outcome | Action | Owner |
|---------|--------|-------|
| New secrets / env vars introduced | Secrets reference (README, CONTRIBUTING .dev.vars template) | software-docs-minion |
| New response headers | API reference, README headers section | software-docs-minion |
| Error response shape changed | OpenAPI error response definitions | software-docs-minion |
| Existing behavior changed | Scan docs referencing changed behavior for stale content | software-docs-minion |
| New publicly accessible endpoint (any) | README endpoint table, OpenAPI spec | software-docs-minion |
| Developer setup dependencies changed | CONTRIBUTING.md setup section | software-docs-minion |

These six rows address every WRL drift item that the current table missed.

#### R2: Add a diff-based trigger alongside the semantic trigger

The outcome-action table should not be the only input to Phase 8 checklist generation. Add a complementary trigger: scan the actual git diff from Phase 4 for documentation-relevant patterns. Specifically:

- New environment variable reads (`process.env.X`, `env.X`, `.dev.vars`) that do not appear in README or CONTRIBUTING.md
- New HTTP status codes in route handlers not present in OpenAPI spec
- New response header definitions (`headers.set`, `headers.append`) not in OpenAPI spec
- New route registrations not in OpenAPI spec
- Modified OpenAPI-documented response structures where the code diverged

This is not a full CI linter -- it is a lightweight heuristic scan that the orchestrator runs during Phase 8 step 1 (checklist generation) by reading the git diff and comparing against documentation file contents. The scan produces additional checklist items alongside the outcome-action table matches.

#### R3: Split OpenAPI verification between Phase 6 and Phase 8

OpenAPI spec accuracy is both a test concern AND a documentation concern. They address different failure modes:

- **Phase 6 (test) concern**: Does the running code match what the spec says? This is contract testing. Tools like Schemathesis, Prism, or `openapi-diff` validate behavioral conformance. If tests exist that validate against the spec, Phase 6 catches behavioral drift. This is the stronger guardrail -- it is automated and deterministic.

- **Phase 8 (docs) concern**: Does the spec document all endpoints, headers, and error shapes that the code produces? This is completeness verification. Phase 6 tests only validate what the spec already declares; they cannot catch *missing* declarations. Phase 8 should scan for undocumented routes, response headers, and error codes.

Recommendation: Add an outcome row "OpenAPI spec exists" with action "Verify spec completeness against route handlers" owned by software-docs-minion. This runs in Phase 8. Separately, recommend (as an ADVISE to test-minion) that projects with OpenAPI specs add contract validation tests to Phase 6.

#### R4: Verify Phase 4 documentation task completeness in Phase 8

When Phase 4 includes a task tagged as documentation work, Phase 8 must not skip based on that task's existence. Instead:

1. Phase 8 checklist generation runs normally (outcome-action table + diff scan).
2. Any checklist items that overlap with the Phase 4 docs task are checked for completion: did the Phase 4 task actually update the relevant files?
3. Unaddressed checklist items become Phase 8 work items, regardless of the Phase 4 task's self-reported status.

This eliminates Pattern 3 ("covered by Task N" when Task N covered a subset).

#### R5: Make Phase 8 checklist generation mandatory even when Phase 8 execution is skipped

This directly addresses Pattern 1 (user-directed skip) and Pattern 5 (cross-PR accumulation). When the user selects "Skip docs" or "--skip-docs":

1. Phase 8 step 1 (checklist generation) still runs.
2. If the checklist is non-empty, display it as an informational note: "Skipped documentation items: [list]. These remain unaddressed."
3. The checklist is written to scratch as normal, creating a record of acknowledged debt.

This costs one lightweight step (no subagent spawning, just the orchestrator evaluating the outcome-action table) and makes the debt visible without forcing the user to act on it.

#### R6: Add "Stale content" as a first-class outcome category

The most damaging drift in the WRL audit was not missing documentation -- it was *wrong* documentation. The key rotation warning actively misled deployers. The outcome-action table has no category for "existing documentation contradicts new behavior." Add:

| Outcome | Action | Owner |
|---------|--------|-------|
| Behavior contradicts existing docs | Identify and fix stale content in README, guides, OpenAPI | software-docs-minion |

The trigger for this row is any execution task that changes the behavior of an already-documented feature. The diff-based scan (R2) helps identify this by matching changed function signatures, route handlers, or configuration parsing against documentation references.

### Risks and Concerns

**Risk 1: Checklist inflation.** Adding 7 new outcome rows plus diff-based augmentation could produce noisy checklists where most items are false positives. Mitigation: the diff-based scan should be conservative (only flag clear mismatches, not speculative gaps). The outcome-action table additions are precise categories that match specific artifact types.

**Risk 2: Mandatory checklist on skip adds friction to the skip path.** If the checklist is always generated, users who legitimately want to skip docs (e.g., rapid prototyping) see a list of items they are choosing to defer. This is the intended behavior -- making debt visible -- but could feel nagging. Mitigation: keep the output to a single CONDENSE line with item count, not a full checklist dump. "Documentation items deferred (3): secrets ref, endpoint docs, stale README section."

**Risk 3: Diff-based scan is heuristic and fragile.** Pattern matching git diffs for env var reads, route registrations, and header definitions is language-dependent and imprecise. Mitigation: this is an augmentation to the outcome-action table, not a replacement. False negatives from the diff scan are acceptable because the outcome-action table provides the baseline. False positives are caught by the documentation agent during execution.

**Risk 4: Dual ownership of OpenAPI (Phase 6 + Phase 8) could create confusion.** If both phases touch OpenAPI concerns, responsibilities may overlap or gaps may fall between them. Mitigation: clear separation -- Phase 6 validates behavioral conformance (does the running API match the spec?), Phase 8 validates completeness (does the spec cover everything the API does?). Document this boundary explicitly in the spec.
