## Security Review: harness-iteration-roadmap Execution Plan

### Summary

This plan produces two documentation files and one cross-link update -- no code, no secrets, no APIs, no user input handling, no runtime components. The security surface is minimal. The primary security considerations are: content of the generated documents themselves (do they inadvertently encode insecure patterns into the roadmap?), and the process by which the agent writes files (path targeting, scope creep). Both are low risk given the plan's constraints. Overall risk level: **Low**.

---

### Findings

#### LOW: Roadmap Encodes Credentials as Environment Variables Without Storage Guidance

- **Location**: Phase 3 synthesis, Task 2 prompt -- Issue 1.2 scope bullet: "Environment variable overrides for CI/CD"; Dependencies table: "`CODEX_API_KEY` or saved auth"
- **Description**: The roadmap instructs that API keys for Codex and Aider be handled via environment variables, but the generated document will not include guidance on how those env vars should be stored or injected (secret store vs. plaintext `.env` files vs. shell profile). Users reading the roadmap and implementing the config layer may reach for the most convenient pattern (plaintext `.env` committed to the repo).
- **Impact**: If a developer follows the roadmap naively and stores `CODEX_API_KEY` in a `.env` file that gets committed, that is a credential exposure. This is a documentation-induced vulnerability pattern (OWASP A02, A07).
- **Remediation**: In the Issue 1.2 scope bullets, have the agent add one line: "Environment variable overrides for CI/CD -- document that keys must be sourced from a secret store (e.g., Vault, CI secrets, or a password manager CLI), never from a committed `.env` file." This costs one sentence and closes the gap.

#### LOW: Instruction File Lifecycle Leaves a Cleanup Race Window

- **Location**: Issue 1.1 scope: "Define instruction file lifecycle: write to worktree before invocation, clean up after completion/failure"; Issue 1.3: "Cleanup is part of the adapter lifecycle"
- **Description**: The roadmap defers instruction file cleanup to the adapter lifecycle. If an adapter crashes or is killed mid-execution, the instruction file (containing the full agent system prompt) persists in the worktree. For multi-user or shared environments this leaks system prompt content. The plan mentions a "marker for emergency cleanup" in the Risks table but does not escalate this to an issue.
- **Impact**: System prompt disclosure (analogous to LLM07). For a single-user private repo the impact is low, but the roadmap will be published under Apache 2.0 and used by others who may run in shared or CI environments.
- **Remediation**: The existing Risks table entry is sufficient acknowledgment. No plan change required. The agent writing Task 2 must preserve this risk note verbatim and add a one-line note to Issue 1.1's Notes field: "Instruction files may persist on adapter crash. Write with a `.harness-tmp` suffix and register a cleanup handler on process exit."

#### INFORMATIONAL: Worktree Path Passed to External Processes Without Explicit Validation Requirement

- **Location**: Issue 2.1 scope: "cwd (worktree path)"; Issue 3.2: same pattern
- **Description**: The `workingDirectory` field in `DelegationRequest` will be passed as the working directory to external CLI tools. The plan does not specify that this path must be validated. If routing config or orchestrator ever derives this path from user-provided input without canonicalization, this becomes a path traversal vector (CWE-22).
- **Impact**: Informational at this stage -- the path is generated by the orchestrator from worktree creation, not from raw user input. This is a future implementation concern, not a documentation risk.
- **Remediation**: No change to the plan. Issue 1.1's DelegationRequest type definition should note that `workingDirectory` must be an absolute canonicalized path generated by the orchestrator, not accepted from external input.

#### INFORMATIONAL: Vendor-Specific Model IDs in Published YAML Examples

- **Location**: Task 1 prompt, Change 1, ADVISORY note: "Use current real model identifiers (o3, o4-mini, claude-opus-4-6, gemini-2.5-pro)"
- **Description**: The ADVISORY directs the agent to embed real vendor-specific model identifiers in YAML examples. The document is published under Apache 2.0. This is not a security issue -- noted for completeness as an intentional tradeoff (concrete examples over abstraction). The plan is self-aware about vendor-neutrality requirements and this carve-out is deliberate.
- **Impact**: None.
- **Remediation**: None required.

---

### Recommendations

1. **Add one sentence to Issue 1.2 scope** about secret store requirements for API key env vars. This is the only change that materially reduces downstream security risk induced by the documentation.

2. **Preserve the instruction file cleanup risk note** from the Risks table when Task 2 is executed. The plan already identifies this; the agent must not silently drop it when writing the roadmap. Add a one-liner to Issue 1.1 Notes.

3. No other changes are required. The plan is well-scoped, tasks are limited to documentation edits under `docs/`, no secrets are referenced or handled, and the sequential approval gate on Task 1 provides an adequate checkpoint before the roadmap is created.

---

**Verdict: APPROVE**

The plan is security-sound for its scope. Both findings are Low/Informational and do not block execution. The single actionable item (secret store guidance in Issue 1.2) is a one-sentence addition that can be incorporated during Task 2 execution without requiring a plan revision.
