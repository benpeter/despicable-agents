ADVISE

- [security-minion]: The behavioral contract delegates validation of `working_directory` to adapter authors without defining what validation means, leaving path traversal and symlink-following attacks unmitigated.
  SCOPE: `DelegationRequest.working_directory` field definition and Adapter Behavioral Contract section
  CHANGE: The contract must specify: (1) canonicalization resolves symlinks before validation, (2) the resolved path must fall within an allowlisted root (project repository root or an explicitly configured base path), and (3) the validation requirement should state "reject requests whose canonicalized path resolves outside the allowed root" -- not just "must be validated as an existing directory."
  WHY: "Validated as an existing directory" is ambiguous. An adapter author implementing `os.path.exists()` satisfies the current spec literally but does not prevent a crafted `working_directory` value from targeting `/tmp`, `~/.ssh`, or any path reachable via symlink. Since adapters invoke external tools (Codex CLI, Aider) in this directory, a traversal-enabled working directory gives those tools write access to arbitrary filesystem locations.
  TASK: 1

- [security-minion]: The contract says task_prompt is passed to the external tool "as-is" with no sanitization guidance, leaving shell command injection unaddressed for adapters that invoke tools via shell interpolation.
  SCOPE: `DelegationRequest.task_prompt` field description and Adapter Behavioral Contract "What adapters must NOT do" section
  CHANGE: Add an explicit prohibition in the behavioral contract: adapters must invoke external tools using argument vector APIs (execv-family, subprocess with list args, child_process.execFile) -- never via shell string interpolation of the task_prompt. The "What adapters must NOT do" section should include: "Pass task_prompt as an interpolated shell string; always use argument vector invocation."
  WHY: If an adapter constructs a shell command like `aider --message "$task_prompt"` and task_prompt contains shell metacharacters (backticks, `$(...)`, `&&`, pipes), the shell will evaluate them. The task_prompt field carries untrusted content from the orchestrator's task queue, which may itself be populated from external inputs. This is a classic CWE-78 vector. Specifying safe invocation patterns at the contract level prevents every adapter from independently getting this wrong.
  TASK: 1

- [security-minion]: The contract excludes the `environment` field to prevent API keys in delegation requests, but provides no guidance on how adapters should handle secrets they need for tool invocation -- leaving adapters to silently inherit and forward the full parent process environment, which may include unrelated credentials.
  SCOPE: "Fields Considered and Excluded" section (`environment` entry) and Adapter Behavioral Contract section
  CHANGE: Expand the `environment` exclusion rationale to include the expected pattern: adapters source their own secrets from the host environment or a dedicated secret store (not from the DelegationRequest), and must not log or surface credential-bearing environment variables in `stderr` or diagnostic output. Add a behavioral contract note: "Adapters must not include credential values in any DelegationResult field."
  WHY: The current rationale ("Security concern without a concrete use case") correctly blocks the field but leaves adapter authors with no guidance. The natural workaround is `subprocess.run(env=os.environ.copy())`, which passes all parent-process secrets to the child tool. If the child tool logs its environment (many do in debug mode), credentials appear in `stderr`, which flows back into the DelegationResult and potentially into orchestration logs. A concrete pattern prevents accidental disclosure.
  TASK: 1

- [security-minion]: Temporary files (`translated_instruction_path`, `raw_diff_path`) have no file permission requirements, allowing world-readable creation that exposes instruction content and diff output to other processes on the host.
  SCOPE: Adapter Behavioral Contract "Lifecycle" and "Error handling" sections; `DelegationResult.raw_diff_path` field description
  CHANGE: (1) Add a requirement that the adapter creates the translated instruction file with permissions restricted to the invoking user (mode 0600 or equivalent). (2) Change `raw_diff_path` cleanup language from advisory ("May be cleaned up") to bounded: "Must be cleaned up within the orchestration session. If the orchestrator does not consume this file within the session, the adapter is responsible for deletion." (3) Specify that temporary files must be created in a session-scoped subdirectory under a secure temp location, not in the working directory itself.
  WHY: Instruction files may contain sensitive system prompt content or agent behavioral overrides. Diff files may contain proprietary code. In shared or containerized environments, world-readable temp files are accessible to any process running as any user. "May be cleaned up" creates indefinite retention with no ownership boundary, which is a data hygiene failure independent of confidentiality classification.
  TASK: 1

- [security-minion]: The `success` boolean gives adapter authors unconstrained latitude to normalize exit codes, with no contract language preventing security-relevant failures (signal termination, permission denied, integrity errors) from being mapped to `success: true`.
  SCOPE: `DelegationResult.success` field semantics and Adapter Behavioral Contract "Error handling" section
  CHANGE: Add a "success normalization constraints" note to the behavioral contract listing failure categories that must always produce `success: false` regardless of adapter-specific semantics: (1) tool process terminated by signal (exit code < 0 or signaled), (2) tool binary not found or not executable, (3) working directory not accessible at invocation time, (4) timeout triggered. Normalize-to-success is only permitted for non-zero exit codes that are documented as non-error conditions by the specific tool (e.g., Aider exit 1 = "no changes needed").
  WHY: Without this constraint, an adapter author normalizing aggressively (e.g., "exit code 126 = permission denied, but we'll call it success") silently masks infrastructure or security failures. The orchestrator proceeds on `success: true` without triggering failure handling. Constraining the normalization space to well-documented tool-specific exit code semantics makes the contract auditable and prevents overly permissive adapters from hiding failures. This also partially addresses the risk that a prompt injection causes the external tool to terminate abnormally -- abnormal termination cannot be masked as success.
  TASK: 1
